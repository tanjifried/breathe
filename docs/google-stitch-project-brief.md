# Breathe Project Brief For Google Stitch

## What is Breathe?

Breathe is a private, self-hosted relational wellness app designed to help couples regulate before conflict escalates, repair after friction, and stay connected through small, steady rituals outside conflict.

It is not a social app, a therapy chatbot, or a gamified mood tracker.
It is a calm, trusted tool for moments when people are most likely to feel flooded, defensive, or misunderstood.

The product spans:

- Android app
- browser extension
- self-hosted server

The Android app should feel like the safest, softest part of the system.

## What is the purpose of this project?

The purpose of Breathe is to reduce relational harm in real moments of tension.

The product exists to help two people:

- notice escalation early
- pause before doing more damage
- regulate their nervous systems
- re-enter with more care
- build connection when things are already calm

Every screen should support emotional safety, clarity, and steadiness.

## What is the aim of the project?

The aim is to create a private relationship support system that feels trustworthy enough to use during real-life conflict.

That means the UI should feel:

- calm
- grounded
- private
- emotionally nonjudgmental
- soft without becoming childish
- structured without becoming clinical or cold

The design should help people slow down, not speed them up.

## What should the product feel like?

The emotional tone should be:

- gentle
- regulated
- intimate
- low-pressure
- restorative
- quiet but confident

It should not feel like:

- a productivity app
- a startup dashboard
- a playful dating app
- a bright wellness influencer brand
- a mental-health crisis app
- a sterile medical interface

## What visual vibe should Stitch match?

Use the existing extension as the visual anchor.

The shared visual language is:

- soft sage and cream surfaces
- muted greens with warm yellow and grounded red accents
- rounded cards and pill controls
- subtle borders
- low-contrast shadows
- gentle gradients instead of loud flat fills
- warm, readable typography
- spacious layouts that feel breathable

The interface should feel like a protected conversation corner, not a control panel.

## What should the UX prioritize?

Prioritize:

- calm over novelty
- legibility over density
- emotional safety over visual cleverness
- clear actions over too many options
- touch comfort over compact layouts
- soft transitions over flashy motion

Users should be able to open the app while stressed and still understand what to do in seconds.

## What should be avoided?

Avoid:

- neon gradients
- AI-looking purple and pink palettes
- overly glossy consumer-fintech visuals
- gamification cues
- trophy or score language
- judgmental copy
- aggressive error states
- dense dashboards
- decorative clutter
- harsh black-and-white contrast

Do not make the product feel like it is measuring relationship performance.

## Who is the product for?

The product is for couples who want a practical, private structure for:

- daily connection
- conflict prevention
- timeout and calming rituals
- post-conflict reflection
- shared relational awareness

They may already care deeply about one another and still struggle in moments of activation.

## What are the core product principles?

### Absolute privacy

All data stays on the user’s own self-hosted setup.
The UI should reinforce that privacy through calm trust cues, not surveillance or analytics language.

### Physiological regulation first

The app is not mainly about insight or reflection in the middle of escalation.
It is about slowing the body down first.

### Offline-first reliability

The app should still feel dependable when the server is unavailable.
Important flows should not look broken just because sync is unavailable.

### Empathetic engineering

Every action should feel like it lowers pressure instead of adding it.

## What are the main features Stitch should design around?

### 1. Pairing and setup

Users create a private account, sign in, generate a pairing code, or join a pairing code.
There is also a debug-only offline path for development.

UI needs:

- welcoming setup flow
- privacy-forward tone
- low-friction forms
- clear distinction between private account setup and couple pairing

### 2. Home

The home screen is the calm command center.
It shows:

- own status
- partner status
- sync state
- entry points to core tools

UI needs:

- strong hierarchy
- immediate readability
- action tiles or cards that feel supportive, not busy

### 3. Status check-in

Users can signal:

- green
- yellow
- red

This is an early signal system, not a mood score.

UI needs:

- emotionally clear color-coded controls
- plain-language meaning for each state
- reassurance that choosing a state is information, not failure

### 4. Calm session

This is a guided self-regulation space for slowing down before conflict spikes.

Current functionality includes:

- start calm session
- countdown
- complete session
- send peace

UI needs:

- breathing room
- timer visibility
- gentle reassurance
- soft focus on one action at a time

### 5. Structured timeout

This is a firmer intervention for high activation.
It creates a real lock/re-entry boundary.

Current functionality includes:

- start timeout
- active countdown
- lock state
- re-entry message after unlock

UI needs:

- more structure than Calm
- visible time boundary
- clear feeling of safety and containment
- no punitive tone

### 6. Voice Studio

Users prepare warm voice prompts while calm so they are available later.

Current functionality includes:

- prompt slots
- selected slot preview
- local recording toggle state

UI needs:

- intimate studio-like feel
- warmth and clarity
- lightweight recording affordances
- less “audio tool”, more “care preparation”

### 7. Conflict Log

Users can review local records of recent Calm and Timeout sessions.

Current functionality includes:

- entry list
- selected-entry details
- mood shift summary
- shared/private status

UI needs:

- reflective rather than analytical tone
- readable chronology
- details that feel private and contained

### 8. Weekly Insights

The product offers a weekly summary of patterns, not a score.

Current functionality includes:

- headline
- top feature used
- average mood
- mood direction
- recommendations

UI needs:

- interpretive calmness
- summary cards
- no judgment
- no competitive language

## What future features should Stitch keep in mind?

The UI direction should leave room for these future modules:

- Micro-Connections
  Daily low-pressure relationship questions and connection rituals

- Guided Talk
  Yellow-zone structured conversation prompts and “I-statement” support

- Shared Logistics
  Private shared lists and micro-calendar to reduce friction from planning issues

- Relational Health Assessments
  Weekly pattern summaries based on check-ins and conflict logs

The overall system should grow without losing its quiet, grounded identity.

## What should the Android app specifically become?

The Android app should feel like a personal refuge that still supports couple coordination.

It should be:

- more immersive than the extension
- more emotionally spacious than the web surfaces
- more soothing than a standard Material app

The app should feel safe enough to open during stress and warm enough to open during calm.

## How should motion and interaction feel?

Motion should be:

- subtle
- slow enough to feel steady
- supportive of focus
- never flashy

Interaction should feel:

- forgiving
- touch-friendly
- calm under pressure

If animation is used, it should suggest breathing, settling, or soft emphasis.

## What should Stitch deliver?

Design upgrades should focus on:

- stronger visual hierarchy
- more polished layouts
- calmer and more premium surfaces
- reusable Compose-friendly component patterns
- screen designs that match the extension’s sage/cream identity
- preserving the existing product mood and purpose

The result should feel like a carefully designed private relationship tool, not a generic mobile template.
